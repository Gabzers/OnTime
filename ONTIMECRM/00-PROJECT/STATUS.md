- [x] **Proposal/client/sale field-name drift fix** (2026-06-24, found via live browser testing): frontend types had drifted from the backend contract in several places, causing values to silently render blank or `0,00 €`:
  - `ProposalListDto`/`ProposalDto.value` → backend sends `proposalValue`. Fixed in `types/api-proposal.ts`, `ProposalDetailPage.tsx`, `ProposalsPage.tsx`, `ClientDetailPage.tsx`.
  - `ProposalVehicleDto.brandName` → backend sends `modelBrandName`; also added the missing `id`/`freeTextModel` fields and made `modelId`/`modelName` optional (free-text vehicles have neither).
  - **Convert-to-Sale modal had no `finalValue` input at all** — it silently sent `undefined`, which the backend's non-nullable `decimal FinalValue` deserialized as `0`. This is why every converted sale showed `0,00 €`. Added the missing field, defaulting to the proposal's value.
  - `ClientDto.activeProposal` — backend never returns this field; removed the dead "Proposta Activa" card from `ClientDetailPage.tsx` Info tab (the data the user expected to see here is in the Propostas tab, which works).
  - `StageHistoryDto` (frontend) had `stageName`/`notes`, but backend sends `fromStageName`/`toStageName`/`toStageColor`/`obs` — the Histórico tab Timeline was rendering nothing but the date. Fixed to show `"{FromStage} → {ToStage}"` + `obs`.
  - `ClientSaleHistoryDto` (frontend) had `value`/`vehicleName`/`createdAt`, but backend sends `finalValue`/`modelName`+`freeTextModel`/no `createdAt`. Fixed the Vendas tab on `ClientDetailPage.tsx` accordingly.
  - Regression test added: `src/__tests__/utils/proposalValue.test.ts`.

- [x] **Stage edit form was a no-op for Active/Final/Won/Lost** (2026-06-24, found via live browser testing): `UpdateStageRequest` (backend) never had `IsActive`/`IsFinal`/`IsWon`/`IsLost` fields at all — the frontend `StagesPage` form collected these toggles but they were silently dropped by JSON binding, and **`IsActive` defaulted to `false`** on every edit (since the frontend didn't send it either), meaning editing *any* stage's name/color silently deactivated it. Fixed: added the fields to `UpdateStageRequest` + `ClientStageService.UpdateAsync`; only one stage can be `IsWon` and one `IsLost` per user — setting either now auto-demotes the previous holder (`STAGE_WON_AND_LOST` 422 if both requested on the same stage). Also: `ClientStageDto.ClientCount` didn't exist on the backend at all (frontend read `stage.clientCount` from nothing, always `undefined`) — the "N clientes" tag rendered blank and, more importantly, `undefined > 0` always evaluated false so the delete button's "has active clients" guard never actually blocked anything client-side (backend still rejected via `STAGE_HAS_CLIENTS`, so no data was at risk, but the UX never warned upfront). Added real `ClientCount` per stage in `StageRepository.GetByUserAsync`. Frontend: `StagesPage.tsx` color field now uses AntD `ColorPicker` (was a raw hex `Input`), added the missing "Ativa" toggle, mutual-exclusion UI for Won/Lost, and a link to Notification Settings for managing each stage's automatic notification templates (which are not editable from this page). 4 new backend tests (`StageConfigFlowTests`).
- [x] **Dashboard "Negócios Quentes" field-name drift** (2026-06-24): `DashboardDto.hotDeals` was typed as `ClientListDto[]` but the backend returns a distinct lean shape (`fullName`, `stageName`, `stageColor`, `lastInteractionAt` — no `temperature`). The "Etapa" column read `currentStageName` (always blank) and the "Temp." column/filter read `temperature` (which doesn't exist on this DTO), so `tempKey(undefined)` resolved to the literal fallback key `'MSG.EMPTY_STATE.DEFAULT'`, rendering "Sem resultados." inside the Temp. cell. Fixed: added `HotDealDto` type, fixed the Etapa `dataIndex`, removed the broken/meaningless temperature filter (hot deals are always `Temperature=Hot` by definition — there's nothing to filter), Temp. column now renders a static Hot tag.
- [x] **VehicleProposalTable: Marca blank when editing an existing proposal** (2026-06-24): when opening "Editar" on a proposal whose vehicle already has a `modelId` but the row's `brandId` was never set (only inferred from `brandIds` when the user has exactly one selected brand), the Marca dropdown showed empty even though Modelo correctly showed the model name. Added a `BrandSelect` wrapper that infers the brand from the model (mirroring `ModelSelect`'s existing pattern) without clearing the model selection.
