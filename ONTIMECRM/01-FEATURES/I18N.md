# Feature: i18n (translations)

Hub: [[OnTimeCRM]] · Schema: [[SCHEMA-CONFIG]] · Roadmap: [[ROADMAP]] (Phase 3).

## Current reality (verified 2026-06-25)
- Translations are served from **hardcoded C# dictionaries** in `I18nController.cs` — `PtPT()` and `EnUS()`. Not from the database.
- `GET /api/i18n?locale=pt-PT` → `{ v: "20260308", locale, map }`, plus `X-I18n-Version` header. Unknown locale falls back to pt-PT.
- The `TranslationEntry` table exists but is **dead** — referenced only by the DbContext/EF config, never read or written. See [[SCHEMA-CONFIG]].
- Frontend: `i18nStore` (Zustand + persist) loads on boot; `useTranslations()` returns a Proxy that returns the key itself if missing (never crashes). Bundled fallback in `i18nFallback.ts` — **pt-PT only**, there's no en-US fallback file; English always depends on the live API call succeeding.
- **English is now a real, complete second language** (2026-06-25 — was previously decorative; the product is now deliberately bilingual, see CLAUDE.md rule 1). ~130 hardcoded Portuguese strings across 15+ pages/components were moved to `t['...']` keys with both locale values. Root cause of "translation looked broken everywhere" was **`App.tsx`'s antd `ConfigProvider`** hardcoding `locale={ptPT}` regardless of the selected language — antd's own internal strings (pagination "X / page", date pickers, etc.) never changed no matter what `i18nStore.locale` was. Fixed: `locale={locale === 'en-US' ? enUS : ptPT}`.
- New helper `src/utils/format.ts` (`tFormat`) for translated strings with positional placeholders (`"{0} model(s)"`) — there was no prior way to interpolate a value into a translated string.
- Stage names, brand names, company names — **tenant data, never UI strings.** A company types its own pipeline stage names (e.g. "Venda") freely; they're stored as-is and shown verbatim regardless of locale, exactly like a client's name. Don't "fix" a literal stage name not changing with the language toggle — that's correct.

## The rule
All UI text comes from `/api/i18n`; never hardcode. See [[CONVENTIONS]].
```tsx
t['LABEL.CLIENT.FULL_NAME']   // ✅
'Nome Completo'               // ❌
```

## Key convention
```
NAV.*  ACTION.*  LABEL.*  ENUM.*  MSG.*  PAGE.*  KPI.*  HINT.*  VALIDATION.*
```

Two keys specifically were missing from **both** locale dictionaries despite a frontend-only fallback existing (`LABEL.NOTIF_SETTINGS.DAILY_DIGEST_TIME`/`DIGEST_ENABLED`) — that's the actual failure mode to watch for: a key present in `i18nFallback.ts` but absent from `I18nController.cs` always shows the pt-PT fallback text regardless of locale, silently.
- **`ENUM.NOTIFICATION_STATUS` labels swapped** (2↔3) vs the enum (`Snoozed=2, Ignored=3`).
- Dead enum keys: `ENUM.TRANSMISSION`, `ENUM.VEHICLE_SEGMENT` (no backing fields).
- Missing keys are invisible (Proxy returns the raw key) — no systematic gap detection. **Manual check that now exists**: compare `GET /api/i18n?locale=pt-PT` vs `?locale=en-US` key counts — see CLAUDE.md rule 1 and [[CONVENTIONS]] for the exact command.

## Phase 3 — rebuild plan (to decide via `/grill-me`)

Decide first: **keep hardcoded dictionaries, or move to DB?**

If DB-backed:
- Add `translation_locales` and wire `translation_entries.locale_id` (see [[SCHEMA-CONFIG]]).
- Seed pt-PT/en-US from the current `I18nController` dictionaries.

Missing-key detection:
```
Proxy misses key → (dev) show "⚠ KEY" → (prod) fallback pt-PT → POST /api/i18n/missing { key, locale }
```

Auto-translation (optional):
```
Missing key logged → exists in base (pt-PT)? → DeepL/LibreTranslate → store (is_reviewed=false)
```

Open decisions: DeepL vs LibreTranslate · runtime vs background · admin review panel · which locales beyond pt-PT/en-US.
